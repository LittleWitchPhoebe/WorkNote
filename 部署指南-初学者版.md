# WorkNote 部署指南（初学者版）

本文档面向第一次接触「GitHub → 阿里云镜像 → Minikube」流程的读者，按步骤说明如何让代码在推送到 GitHub 后，由阿里云自动构建镜像，并在本地 Minikube 中运行与访问。

---

## 第一步：理解整体流程

我们要做的事可以概括为一条链：

```
你在本机写代码
    ↓
git push 到 GitHub
    ↓
阿里云 ACR 发现 GitHub 有更新，按你配置的规则自动构建 Docker 镜像并保存
    ↓
你在本机用 Minikube 从阿里云拉取该镜像并部署
    ↓
在浏览器通过 Minikube 提供的地址访问你的应用
```

**要点：**

- 阿里云镜像仓库已经和 GitHub 仓库（LittleWitchPhoebe/WorkNote）关联。
- 阿里云会**自动**根据「构建规则」在代码变更时构建镜像，无需你手动在阿里云控制台点「构建」。
- 你本地只需要：能 push 到 GitHub、能跑 Minikube、能执行 `kubectl`。

---

## 第二步：准备本机环境

请确保已安装以下工具（若已安装可跳过对应小节）。

### 2.1 Git

- 用途：把代码推送到 GitHub，触发阿里云构建。
- 安装（macOS）：`xcode-select --install` 或从 [git-scm.com](https://git-scm.com/) 安装。
- 验证：终端执行 `git --version`，能输出版本号即可。

### 2.2 Docker

- 用途：本地可选用来验证镜像；Minikube 底层也会用到 Docker（或其它驱动）。
- 安装：安装 [Docker Desktop for Mac](https://docs.docker.com/desktop/install/mac-install/)。
- 验证：`docker --version`。

### 2.3 Minikube

- 用途：在本地跑一个「迷你」Kubernetes 集群，用于部署你的镜像。
- 安装（macOS，用 Homebrew）：  
  `brew install minikube`
- 验证：`minikube version`。

### 2.4 kubectl

- 用途：和 Minikube 里的 Kubernetes 通信，用来部署、查看服务。
- 安装（macOS）：`brew install kubectl`
- 验证：`kubectl version --client`。

---

## 第三步：在阿里云 ACR 配置「自动构建」

只有配置好「代码变更时自动构建」，push 到 GitHub 后阿里云才会自动构建镜像。

1. 登录 [阿里云容器镜像服务控制台](https://cr.console.aliyun.com/)。
2. 找到你的个人实例 / 命名空间，进入仓库 **test017/test**（即你提供的公网地址对应的仓库）。
3. 在仓库页面找到 **「构建」** 或 **「构建配置」** 相关入口。
4. 新增或编辑一条构建规则，建议配置为：
   - **代码源**：已绑定的 GitHub 仓库（LittleWitchPhoebe/WorkNote）。
   - **分支**：例如 `main`（和你实际推送的分支一致）。
   - **构建上下文路径**：填 **`app`**（因为本项目的 Dockerfile 和页面在 `app/` 目录下）。
   - **Dockerfile 路径**：  
     - 若控制台要求「相对于仓库根」：填 **`app/Dockerfile`**；  
     - 若要求「相对于构建上下文」：填 **`Dockerfile`**。
   - **镜像标签**：可设为 **`latest`**，或按需使用「Git 提交 ID」等变量。
5. 保存后，可手动触发一次构建做测试，或等第四步 push 代码后自动触发。

同时建议在控制台 **「访问凭证」** 里确认登录用户名和密码（本指南后面若使用私有仓库拉取会用到）。

---

## 第四步：把代码推送到 GitHub

在项目根目录（能看到 `app/`、`k8s/` 的目录）执行：

```bash
git add .
git commit -m "添加容器应用与 K8s 部署"
git push origin main
```

如果你的默认分支名不是 `main`，请把上面命令里的 `main` 改成你的分支名（需和阿里云构建规则里配置的一致）。

推送成功后，到阿里云 ACR 该仓库的「构建」或「构建记录」里查看，应会出现一次新的构建任务；等状态变为「成功」后再进行下一步。

---

## 第五步：启动 Minikube 并确认能访问集群

在终端执行：

```bash
minikube start
```

首次启动会下载镜像和依赖，可能需要几分钟。看到类似 “Done! kubectl is now configured to use "minikube"” 即表示成功。

可选验证：

```bash
kubectl get nodes
```

应能看到一个 `minikube` 节点且状态为 `Ready`。

---

## 第六步：若镜像仓库为「私有」，先配置拉取密钥

若你的阿里云 ACR 仓库是**私有**的，Minikube 里的 Kubernetes 需要凭据才能拉取镜像。在**部署应用之前**执行（只需做一次）：

```bash
# 先创建命名空间
kubectl create namespace worknote

# 创建拉取镜像用的密钥（把 <你的ACR密码> 换成真实密码）
kubectl create secret docker-registry aliyun-acr-secret \
  --namespace=worknote \
  --docker-server=crpi-tdt2zc9s3n24ef5b.cn-hangzhou.personal.cr.aliyuncs.com \
  --docker-username=aliyun1422332421 \
  --docker-password=<你的ACR密码>
```

然后**编辑项目里的 `k8s/deployment.yaml`**，找到 `imagePullSecrets` 两行，去掉注释，例如：

```yaml
spec:
  imagePullSecrets:
    - name: aliyun-acr-secret
  containers:
    ...
```

若仓库为**公开**，可跳过整段第六步，并保持 `deployment.yaml` 里 `imagePullSecrets` 为注释状态。

---

## 第七步：在 Minikube 中部署应用

在项目根目录执行：

```bash
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml
```

每条命令成功时会提示 `created` 或 `configured`。

检查 Pod 是否在运行：

```bash
kubectl get pods -n worknote
```

状态为 `Running` 且 READY 为 `1/1` 即表示部署成功。若长时间不是 Running，可用下面命令看原因：

```bash
kubectl describe pod -n worknote -l app=worknote
```

---

## 第八步：在浏览器中访问应用

任选一种方式即可。

**方式 A（推荐）：让 Minikube 打印访问 URL**

```bash
minikube service worknote -n worknote --url
```

终端会输出一个地址，例如 `http://192.168.49.2:30080`，在浏览器中打开即可。

**方式 B：自己拼地址**

```bash
minikube ip
```

记下输出的 IP，在浏览器访问：`http://<该IP>:30080`。

若能看到「WorkNote · 容器已成功运行」的页面，说明从 GitHub → 阿里云镜像 → Minikube 的整条链路已打通。

---

## 第九步：之后每次更新怎么操作

1. **改代码**（例如改 `app/index.html` 或 `app/Dockerfile`）。
2. **推送到 GitHub**：  
   `git add . && git commit -m "更新内容" && git push origin main`
3. **等阿里云构建完成**（在 ACR 控制台看构建记录）。
4. **让 Minikube 用新镜像**：  
   ```bash
   kubectl rollout restart deployment/worknote -n worknote
   ```
5. 再次用第八步的地址访问，即可看到更新后的效果。

## 常见问题速查

| 现象 | 可能原因 | 建议操作 |
|------|----------|----------|
| push 后阿里云没有自动构建 | 构建规则未保存或分支/路径不对 | 在 ACR 检查构建规则：分支是否为 `main`，构建上下文是否为 `app`，Dockerfile 路径是否正确 |
| 阿里云构建失败 | 构建上下文或 Dockerfile 路径错误 | 构建上下文必须是 `app`；Dockerfile 路径相对于上下文为 `Dockerfile`，或相对于仓库根为 `app/Dockerfile` |
| `kubectl apply` 后 Pod 一直 ImagePullBackOff | 私有仓库未配拉取密钥或密钥错误 | 按第六步创建 `aliyun-acr-secret`，并在 `deployment.yaml` 中启用 `imagePullSecrets` |
| 访问不到页面 | Minikube 未启动或端口不对 | 执行 `minikube status` 确认 Running；用 `minikube service worknote -n worknote --url` 获取正确地址，或确认使用 30080 端口 |
| 想先在本地验证镜像再推 GitHub | - | 在项目根目录执行：`docker build -t crpi-tdt2zc9s3n24ef5b.cn-hangzhou.personal.cr.aliyuncs.com/test017/test:latest ./app`，再 `docker run -p 8080:80 <上面镜像名>`，浏览器访问 `http://localhost:8080` |

## 相关文件说明

- **app/**  
  - `index.html`：容器内展示的页面。  
  - `Dockerfile`：用于构建镜像，阿里云和本地 `docker build` 都会用到。
- **k8s/**  
  - `namespace.yaml`：创建 `worknote` 命名空间。  
  - `deployment.yaml`：定义用哪个镜像、拉取密钥（若私有）、资源等。  
  - `service.yaml`：用 NodePort 暴露 30080，便于在 Minikube 中访问。
- **DEPLOYMENT.md**：面向已有基础的读者，偏技术摘要的部署说明。

若某一步与你的实际界面不一致（例如阿里云控制台改版），可结合 [阿里云 ACR 文档](https://help.aliyun.com/product/60716.html) 对照操作；流程顺序不变：**配好自动构建 → push 代码 → 等构建成功 → Minikube 部署 → 用 URL 访问**。
